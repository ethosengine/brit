# Parity Ralph Loop — Design

**Date:** 2026-04-22
**Branch:** `gix-brit`
**Status:** Scaffolding complete; first pilot run pending

## Goal

Close `git` ↔ `gix` feature parity one command at a time via a parameterized ralph-wiggum loop. Each invocation handles exactly one `$CMD` (e.g., `push`) and drives it from `absent` → `present`. The loop emits a completion promise only when a separate judgment agent (`gix-steward`) verifies the claim.

Non-goals: feature prioritization (operator picks next command), automated upstream PRs (human-gated handoff), brit-vision covenant work (paused pending parity).

## Architecture

### Three agents, three roles

| Agent | Model | Role | Invoked |
|---|---|---|---|
| `gix-architect` | sonnet | Design, C→Rust translation, implementation | Every iteration (the working agent) |
| `gix-steward` | opus | Completion gate, tie-break, deferral adjudication | Three specific moments only |
| `gix-runner` | haiku | Mechanical tasks: matrix cargo check, grep, scaffold-from-template | On demand, from architect or steward |

### Branch separation

- **`gix-main`** — pristine mirror of upstream gitoxide. Never touched by the loop. Upstream PRs are repackaged off `gix-main` by a human.
- **`gix-brit`** — fire-at-will dev branch. Loop commits freely here. `.expect("why")` allowed; `.unwrap()` never. Cross-crate commits fine. No split-per-concern required.
- **`main`** — brit covenant vision. Paused pending parity.

### Verdict model (per-row parity check)

`expect_parity <mode> -- <args>` runs the same `<args>` through `git` and `gix`, reports match per mode:

- **`effect`** (default) — exit-code match required. Output diff shown but not fatal. For UX flags, progress, wording.
- **`bytes`** — exit-code AND byte-exact stdout/stderr match. For scriptable output: `--porcelain`, `--format=*`, tooling-consumed outputs.

### Scope granularity

One `$CMD` per loop invocation. Inside a command: one `it "..."` block per flag/mode. Scenarios within a flag use nested `(with "...")` blocks per existing journey-test idiom. Internal-plumbing parity (e.g., `send-pack` within `git push`) is **not** in scope for the porcelain loop — it falls out as a dependency when the architect needs a primitive that doesn't exist.

## Artifacts

### Durable state (read every iteration, written across iterations)

- `docs/parity/commands.md` — top-level matrix. git cmd × gix cmd, status (absent / partial / present / deferred). Seeded by hand; full enumeration via `etc/parity/enumerate.sh` later.
- `tests/journey/parity/<cmd>.sh` — per-command parity journey test. One `it "..."` block per flag. Uses Sebastian's existing harness (`title`, `when`, `with`, `expect_run`) plus the new `expect_parity` helper.
- Git log — commits themselves are durable state; every iteration commits.

### Tooling

- `tests/helpers.sh` — `expect_parity` helper added (46 new lines).
- `tests/parity.sh` — wrapper for running a single parity file. Bypasses full `just test` / `tests/journey.sh` (which would trip on pre-existing unrelated failures).
- `etc/parity/prompt.md` — immutable ralph prompt, `$CMD`-substituted per invocation.
- `etc/parity/enumerate.sh` — stub; fills in when we scale past hand-curated matrix.

### Reference

- `vendor/git/` — git upstream submodule at `Git 2.54`. The loop prompt directs the architect to read `vendor/git/builtin/$CMD.c` + `vendor/git/Documentation/git-$CMD.txt` before writing Rust. Translate invariants, not C idioms.

## Decisions (brainstorming sessions 2026-04-22)

### D1 — Parameterization by command
Loop is per-`$CMD`, not a single "parity everything" run. Natural terminal state (flag tree is finite); operator picks order; parallel runs possible.

### D2 — Layered verdict model
Option C from the brainstorm: behavioral equivalence by default, byte-exact where caller declares scriptable flags. Matches git's own test-suite posture (strict on `--porcelain`, looser on UX).

### D3 — Extend Sebastian's journey-test harness
Option C′ from the brainstorm: per-command `.sh` file in `tests/journey/parity/`, sourced via new `tests/parity.sh` wrapper. Reuses `title`/`when`/`with`/`it`/`expect_run`. One new helper (`expect_parity`). No parallel new directory structure.

### D4 — gix-brit discipline = fire-at-will
Cross-crate commits fine. `.expect("why")` allowed; `.unwrap()` still prohibited. The cleanup punch list for upstream handoff is generated by clippy (`clippy::unwrap_used` etc.) on a later pass.

### D5 — SHORTCOMINGS.md is historical context, not deferral whitelist
Most entries there are closeable ("unfinished," not "forbidden"). Loop defaults to ambitious: `DEFER-LEGITIMATE` requires a **hard system constraint** or explicit operator approval. The steward enforces this.

### D6 — Upstream handoff is human-gated
No automation. When a command closes on gix-brit and is worth upstreaming, a human cherry-picks from gix-brit onto a branch off gix-main, reformats to Sebastian-spec (split per crate for breaking, proper conventional commits, full error-chain), and PRs to GitoxideLabs/gitoxide.

### D7 — CLAUDE.md blanket coverage
71 CLAUDE.md files placed across the workspace: root (rich), 3 directory-purpose (docs/plans, etc/plan, tests), 67 crate-level (template: tier, error crate, notable-if-non-obvious). Token cost per invocation is bounded because Claude Code loads hierarchically — only the path from root to CWD.

## Pilot

**First command: `git push`.** Chosen because:
- Entirely absent from gix (no `Push` in `Subcommands` enum) — full scaffolding exercise
- High-value user feature, tracked as unchecked in root `README.md` parity table
- Touches plumbing layers likely to expose other gaps (protocol, pack, refs)

Invocation:
```bash
CMD=push
/ralph-loop "$(sed "s/\$CMD/$CMD/g" etc/parity/prompt.md)" \
  --max-iterations 40 \
  --completion-promise "PARITY-git-$CMD"
```

Expected first iteration: scaffold `tests/journey/parity/push.sh` + `src/plumbing/options/push.rs` + dispatch arm + update matrix row to `partial`. Commit. Iteration 2+: close flags one at a time.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Ralph games the completion promise | `gix-steward` gates it; structured verdict with evidence; rejects if `it` blocks only run `gix`, not both; flags bytes-vs-effect mismatches |
| Pre-existing journey-test failures halt the parity test | `tests/parity.sh` sources only the single parity file, not the full suite |
| Architect hits an ambiguous design choice | Steward tie-break at moment #2 |
| Architect gives up on a hard row | Steward adjudication at moment #3; deferral requires hard constraint or operator approval — no "we'll come back" deferrals |
| Cross-crate commits make upstream cherry-pick hard | Accept it. Upstream handoff is human-gated and repackages by design |
| gix-brit diverges too far from gix-main | Periodic rebase-onto-gix-main by operator; the more we drift, the more painful upstream handoff gets. Loop should prefer additive changes to existing crates over novel crates where possible |
| Iteration budget exhausted without closure | Graceful halt; state persists in parity `.sh` file + git log; operator resumes |

## Open questions (for later, not blocking pilot)

- **Q1.** When `git push` needs `send-pack` plumbing that doesn't exist in gix-protocol, does the loop scaffold it inline or escalate to operator? Current plan: escalate. Revisit after pilot.
- **Q2.** How to detect "this flag actually exercises the code path" vs "it parses but is a no-op"? The steward's adversarial checks catch some cases; some might need a dedicated assertion style.
- **Q3.** When `docs/parity/commands.md` grows past ~30 rows, hand-maintenance breaks down. Stand up `etc/parity/enumerate.sh` at that point.
